6. Key Architectural Outcomes

Sentinel remains the trust gateway before Platform Services.

Platform Services remain the governed runtime foundation.

Guardian remains the trusted digital companion and judgement entity.

UXP remains the presentation and state-visualisation layer.

Guardian Orb must visualise real observable platform state rather than simulate intelligence.

GIA should be treated as a local-first observability and instrumentation capability.

No new governance architecture was required before implementation could begin.

7. Guardian Instrumentation Agent (GIA)

The session refined GIA as the platform observability layer. GIA should begin locally by observing engineering workstation and platform state before wider platform telemetry is introduced.

Initial focus: CPU, memory, storage, process/service health and local engineering environment state.

GIA observes and publishes state. It does not own decisions, policy enforcement or authoritative platform state.

GIA should precede Guardian Orb implementation because the Orb requires observable state to visualise.

8. Guardian Orb and UXP

The Guardian Orb was confirmed as a UXP composition layer and visual representation of platform state. It should consume GIA or other authoritative observable state rather than duplicate state ownership.

The Orb is not a policy engine.

The Orb is not the system of record.

The Orb grows with the platform and should not present unimplemented capabilities as live functionality.

9. Resilience Before Disablement

The Programme Sponsor approved the resilience principle that failure should disable only the affected capability, not the platform. Where resilience has been engineered, the platform should attempt recovery or fall back to the best available local or reduced capability before disabling the function entirely.

Cloud capability failure should fall back to local capability where available.

External provider failure should fall back to alternate provider or local basic function where safe.

Sentinel remains the deliberate fail-closed exception for external trust and connectivity.

10. AIEMS Process Corrections and Lessons

A major theme of ESR-0011 was correcting assumption drift. When uncertainty arose, the correct AIEMS behaviour was to verify through the repository and PBK-0001 rather than rely on conversational memory.

GitHub connector capability must be tested, not assumed unavailable.

WP0 / PBK-0001 must be re-read when process uncertainty arises.

Controlled artefacts must be searched before proposing new artefacts.

Historical artefacts support rationale but do not override controlled artefacts.

These lessons were behavioural applications of AIEMS, not evidence that AIEMS itself required governance changes.
