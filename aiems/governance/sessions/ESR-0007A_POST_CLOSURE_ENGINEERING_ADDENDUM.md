# 2. Executive Summary

Following formal closure of [[ESR-0007_ENGINEERING_SESSION_REPORT|ESR-0007]], additional engineering discussion produced significant strategic recommendations affecting future programme direction.

These recommendations are intentionally captured in ESR-0007A as a separate post-closure addendum. This preserves repository baseline integrity by avoiding reopening the accepted ESR-0007 Engineering Session Report, [[RBL-0008_REPOSITORY_BASELINE|RBL-0008]], [[PST-0001_PROGRAMME_STATUS|PST-0001]] or related closure artefacts.

ESR-0007A becomes the primary strategic handover into ESR-0008. It records product engineering direction, recovery work, subsystem evaluation expectations and future planning themes without creating implementation authority by itself.

---

# 3. Engineering Observations

The following engineering observations were accepted after ESR-0007 closure:

- AIEMS has reached sufficient maturity to support sustained JARVIS product engineering.
- Future engineering effort should primarily focus on JARVIS product capability.
- AIEMS improvements should occur only where they materially improve product engineering, repository confidence, validation quality or controlled delivery.

These observations do not reduce the authority of AIEMS. They clarify that governance should now enable product progress rather than become the dominant engineering output.

---

# 4. Product Engineering Direction

The Programme Sponsor approved the strategic transition from governance-led engineering to sustained JARVIS product engineering.

The accepted operational baseline in [[PCB-0001_PRODUCT_CAPABILITY_BASELINE|PCB-0001]] establishes the current product starting point. Future JARVIS work should build from that baseline, informed by [[RPCA-0001_REPOSITORY_PRODUCT_CAPABILITY_ASSESSMENT|RPCA-0001]], [[JARVIS_PRODUCT_ARCHITECTURE|JARVIS Product Architecture]] and [[JARVIS_CAPABILITY_READINESS_MATRIX|JARVIS Capability Readiness Matrix]].

This transition does not bypass AIEMS. It applies AIEMS to product delivery with proportionate governance, repository-first validation and explicit authority boundaries.

---

# 5. ESR-0008 Direction

ESR-0008 should become a Product Development Planning Session.

Primary objective:

Evaluate candidate technologies and architectures across all major JARVIS subsystems before significant implementation begins.

ESR-0008 should avoid premature subsystem implementation. Its value should be in comparing options, recovering product intent, selecting appropriate technology directions and preparing coherent implementation packages.

---

# 6. Product Vision Recovery

A structured review of historical Engineering Sessions is approved for ESR-0008 planning consideration.

Purpose:

Recover:

- product ideas;
- recommendations;
- architecture;
- subsystem proposals;
- engineering observations.

The recovered material should be compared against the GitHub repository and controlled artefacts.

Each recommendation should be classified as:

| Classification | Meaning |
|----------------|---------|
| Implemented | The recommendation is already present in executable repository evidence or accepted artefacts. |
| Represented | The recommendation is represented conceptually in architecture, baseline, readiness or planning artefacts. |
| Backlog | The recommendation is recorded or should be considered for governed backlog handling. |
| Architecture | The recommendation belongs in future architecture definition before implementation. |
| Deferred | The recommendation is recognised but intentionally held for later consideration. |
| Missing | The recommendation is not represented in the repository and may require future action. |

Where appropriate, ESR-0008 should recommend repository improvements without directly modifying unrelated artefacts or promoting unapproved scope.

# 7. JARVIS Subsystem Evaluation

ESR-0008 should evaluate candidate technologies and architectures for the following JARVIS subsystems and supporting engineering surfaces:

| Area | Evaluation Intent |
|------|-------------------|
| Runtime | Identify suitable runtime architecture and operating model for JARVIS capability growth. |
| Experience | Define how the user experience should evolve as capabilities mature. |
| GUI Platform | Compare GUI technology options before significant interface investment. |
| Knowledge | Assess how JARVIS should use governed repository and product knowledge. |
| OSE | Determine how Organic Semantic Enhancement supports repository navigation and JARVIS knowledge use. |
| Obsidian | Assess Obsidian as an Engineering Knowledge Workspace and graph inspection layer. |
| Repository Awareness | Define how JARVIS should inspect and reason over repository state under approved controls. |
| Engineering Assistant | Define how JARVIS should support engineering activity and package generation. |
| Memory | Evaluate persistent, consent-aware and domain-aware memory concepts before implementation. |
| Sentinel | Prepare future monitoring, awareness and supervisory architecture. |
| Guardian | Prepare permission, safety, approval and protection architecture. |
| Voice | Evaluate voice input and speech interaction options. |
| Vision | Evaluate visual understanding and document/image interpretation options. |
| Workflow | Define product and engineering workflow support requirements. |
| Plugins | Evaluate future extensibility and controlled plugin capability. |
| Monitoring | Define service, runtime and operational monitoring expectations. |
| Internet | Evaluate controlled internet capability under future Sentinel and Guardian governance. |
| Provider Architecture | Define provider independence before external AI provider integration. |

This evaluation should be evidence-led and should not assume implementation authority.

# 8. GUI Philosophy

The following engineering principle is accepted:

The GUI evolves as JARVIS capabilities mature.

The interface visualises capability.

The interface does not drive capability.

Technology selection should occur before significant GUI investment. GUI work should therefore follow product capability decisions, subsystem evaluation and architecture selection rather than attempting to define capability through interface design alone.

By completion of ESR-0009, JARVIS should actively assist engineering in a governed and repository-aware manner.

# 12. AIEMS Working Practice

Repository-First Engineering Protocol now applies equally to:

- historical session recovery;
- product vision recovery;
- architecture selection;
- subsystem evaluation;
- engineering recommendations.

This means future recommendations should be compared against repository evidence before being accepted as missing, deferred, architectural or backlog candidates. Repository evidence remains authoritative over conversational memory.
